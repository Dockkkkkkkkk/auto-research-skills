# 自动研究循环设计

## 目的

这个 skill 封装的是一种通用 agent 研究模式：

1. 先约束环境
2. 再定义记分卡
3. 用短轮次做可比较实验
4. 基于证据保留或回退
5. 保留完整决策轨迹

它更像“运行一个研究系统”，而不是“临时想一个聪明办法”。

## 核心模式

### 1. 先有 scorecard，再做开放探索

每个长周期任务都应先定义 scorecard。最低限度包括：

- Objective
- Primary metric
- Secondary guardrails
- Baseline
- Action surface
- Frozen surface
- 单轮预算
- 停止和审批条件

如果没有这些，agent 很容易把漂移误判成进展。

### 2. 保持 evaluator 稳定

尽量让评估器在各轮之间保持稳定。如果每一轮都改评估规则，实验就失去可比性。

强评估器示例：

- 固定人群切片下的转化率
- 固定预算下的有效线索成本
- 固定测试集下的缺陷复现率
- 固定 rubric 下的答案质量
- 固定检查清单下的研究覆盖度

弱评估器示例：

- “感觉不错”
- “看起来有前景”
- 每轮都改 judge 标准

### 3. 收窄 action surface

不要让 agent 一次性改所有东西。应明确它允许变化的部分，例如：

- prompt
- 文案
- 实验配置
- 用户分层
- 工作流分支逻辑
- 某个边界清晰的模块
- 调研与信息采集策略

同时明确冻结面，例如：

- 主指标或 judge
- 预算上限
- 合规边界
- 目标用户
- 数据切片
- 对外接口

### 4. 小而可回退的实验轮次

每一轮都应足够便宜，便宜到可以果断丢弃。推荐：

- 一轮只验证一个主要假设
- 改动尽量小
- 显式记录结果
- 显式做 keep / discard 判断

这才会让长周期 agent 行为可解释。

## 决策词汇

建议统一使用以下状态：

- `keep`：提升了 frontier，成为新的基线
- `discard`：不如基线，或收益不值复杂度
- `retry`：实验执行失败，应干净重跑
- `escalate`：遇到风险、权限、歧义或审批阻塞

不确定时，优先 `discard`，不要把推测性的复杂度带到下一轮。

## 安全和治理

以下场景应显著加强审批：

- 花钱
- 联系外部人员
- 对外发布内容
- 修改生产系统
- 访问敏感数据
- 产生法律、金融、医疗类表述

这些场景里，agent 仍然可以继续研究和设计实验，但执行动作应被明确 gated。

## 推荐节奏

单轮通常包括：

1. 假设
2. 最小干预
3. 执行
4. 指标记录
5. 简短结论

每 3-5 轮：

1. 更新 frontier summary
2. 清理失败分支
3. 重排下一批假设

## 适配提示

### 编码任务

- 指标可以是测试通过率、延迟、失败率、内存、基准分数
- Action surface 可以限制为单模块、单类配置、单类优化

### 增长或盈利实验

- 指标可以是有效线索、净收入、ROI、激活、留存
- Guardrails 包括预算、合规、反骚扰、品牌风险
- 审批点包括付费投放、外呼、账号注册、公开发布

### 调研与分析

- 指标可以是覆盖率、可信度、冲突消解程度、决策可用性
- 冻结面通常是评估 rubric 和来源质量门槛

### 流程优化

- 指标可以是周期时间、缺陷率、人工触点数、交接延迟
- Action surface 通常只允许改某一个阶段
